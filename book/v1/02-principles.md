# 02 · Principles

The team's reusable principles, in narrative form. The authoritative list (with exact wording and provenance) is `LESSONS.md` — read it. Here's what each one *means in practice* and why it exists.

## Process
1. **Understand before shipping.** Think one step ahead about the *real* need instead of literally executing each instruction and iterating into the answer. _(from 2026-05-31)_
2. **Deliver where it will actually be seen.** If the request hints at a destination (a blog, GitHub is open), publish/push it — or ask first. Don't finish only locally and stop. _(2026-05-31)_
3. **Don't guess what you can verify.** Tools exist — `date`, read the file, run the test, check `git`. A confident guess costs more than a quick check. *This is the team's most-repeated lesson — it relapsed and was caught more than once (see chapter 04).* _(2026-05-31)_

## Taste-driven work (design / format / content)
4. **Ask 2–3 short questions before starting.** Where will it live? What look? Any reference? Don't pick a dev default and rework later. _(2026-05-31)_
5. **Get a reference image as early as possible.** One image beats three paragraphs and skips guessing rounds. _(2026-05-31)_
6. **Less but real.** Cut everything non-essential to the core. Plainness is intent, not laziness. _(2026-05-31)_

## Communication
7. **Direct, not rude.** Critique can be hard — give data and a way forward. Critique to build, not to bury. _(2026-05-31)_
8. **Praise needs evidence.** Point to what and why; score by dimension (design 8 / content 5), not one rounded average. _(2026-05-31)_

## Audience & language
9. **Match the artifact's language to its actual reader.** AI-facing files (`CLAUDE.md`, `LESSONS.md`, this book) → English. Human-facing (the diary) → Thai. _(2026-05-31)_

## Systems & maintenance
10. **One fact, one file — placed by when it's needed.** Each fact in exactly one place; cross-reference, never copy. *Which* file is decided by load-time: what's needed every session before any work → always-loaded `CLAUDE.md`; what's read only while working → `LESSONS.md`, on demand. _(2026-05-31, refined 2026-06-04)_
11. **Don't fall in love with a system before it's proven.** A convention built from 1–2 data points is a hypothesis dressed as a conclusion. Forms harden fast — templates, rituals, even self-criticism become reflex — so keep them provisional and vary them on purpose until use proves them. _(2026-06-01)_

---
**Provisional by design:** new principles are hypotheses until they recur across 2+ sessions. The list is kept deliberately lean — before adding one, check it doesn't fold into an existing one (see `LESSONS.md` → "How to use this file"). A list too long to hold in your head is one nobody reads.
